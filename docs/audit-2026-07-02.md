# Repo Audit — 2026-07-02

> Point-in-time audit (requested by Nicolas): project health, process evaluation, anti-patterns,
> inefficiencies, doc staleness, and a benchmark against the ROM-hacking / indie-tactics field.
> Delete or archive this file once its findings are turned into issues.

## Verdict (TL;DR)

The project is in **unusually good shape for what it is** — a solo, AI-driven, decomp-native FE8
hack. Velocity is real (v0.1.0 shipped; Ch0–Ch2 done, Ch3 designed+written, in ~6 working days of
commits), the feature-flow process demonstrably fixed the red-main problem, and several pieces of
infrastructure (mGBA memory-asserted playtest harness, mock-ROM CI, generated-doc freshness gates)
are **top-of-field — better than any shipped FE hack we could find**.

The three real problems, in order of cost:

1. **`build_campaign.py` is a 5,205-line monolith with copy-pasted per-chapter injectors** — the #1
   threat to Ch4–Ch8 velocity. Fix *before* building Ch3's map (#40).
2. **Chapter YAML is not actually the source of truth for chapter structure** — the injectors are.
   Schema drift across ch00–ch03 proves it (no two built chapters express deployment the same way).
3. **Docs are accumulating entropy faster than the drift-guards catch** — HANDOFF.md is ~65%
   durable reference parked in a "live state" doc; contradictions exist (7 vs 8 vs "10 cast",
   #39 vs #65, Baxby Ch1 vs Ch2); `engine/` + `data/homebrew` + `data/srd` are dead scaffolding
   that contradicts a locked ADR.

Everything else is polish: CI can be ~2 min faster (cache agbcc), 9.2 MB of demo GIFs bloat the
pack, 8 merged branches need deleting (#103), 19 issues are stale, and PR review leaves no
auditable artifact.

## 1. What's working (keep doing this)

- **Feature-flow fixed red-main — with receipts.** All 23 CI failures on record were *direct
  pushes to main* (incl. a 21-run red streak on 2026-06-19). Since PR-based flow took hold
  (~06-24): **100% pass over the last 60 runs**. Median CI duration 3.2 min. The process change
  paid for itself.
- **`check.py` as a single drift-guard** shared by CI, pre-commit, and `make check` — including
  `check_generated_indexes_fresh` (CHAPTERS/CLASSES.md regenerated in-memory and diffed) and the
  engine-agnosticism grep. Docs-as-data with *enforcement* is very rare; it's working.
- **`tools/inject/engine_hooks.py` is the best-architected code in the repo**: pure string
  transforms, idempotency guards, `count == 1` form assertions that fail loudly on decomp drift,
  cycle-free `decomp.py` seam, and every patch documents *why*. This is the model for the rest.
- **Mock-ROM CI** (16 MB of `/dev/urandom` as the base ROM) solves the copyright problem with no
  secrets. Clean.
- **`verify_text.py`** — reimplementing the ROM's Huffman text decoder as a no-emulator regression
  gate is exactly the right kind of cheap dynamic check.
- **ADR discipline in `decisions.md`** — dated entries with explicit supersede chains.
- **The mGBA Lua harness** (memory-asserted win/lose/softlock classification, checkpoint states,
  anim capture) — see §4: essentially unprecedented for an FE hack.

## 2. Anti-patterns & inefficiencies (ranked by cost)

### 2.1 The chapter-injector copy-paste machine (HIGH — fix before #40)
- `build_campaign.py`: **5,205 LOC, 94 top-level functions**, owning portraits, names, icons,
  stats, map sprites, reskins, battle anims, tilesets, platforms, title theme, backgrounds,
  montage, and both chapter injectors.
- `inject_ch01` (~895 LOC) and `inject_ch02` (~600 LOC) are **copy-paste siblings** — each
  re-defines near-identical nested helpers (`cut_fid`, `stage`, `split_beats`, map-copy
  boilerplate); ch02's comments say "cf. inject_ch01" 15+ times. **51 module-level
  `CH0x_`/`PROLOGUE_` constants** add a per-chapter global tax.
- At the current pattern, Ch3–Ch8 ≈ **~4,000 more LOC of near-duplicates**, each divergence a
  future bug. Extract `inject_chapter(campaign, chapter_yaml, host_slot, ...)` with shared
  helpers; keep per-chapter YAML as the only per-chapter artifact.
- Same disease in Lua: `tools/playtest/harness.lua` (2,862 LOC, ~90 functions) mixes primitives
  with 6+ scenario drivers. (Lower priority — its likely change is "add a scenario", which is
  cheap; split only when it actually hurts.)
- Bonus cost: `build_campaign.py` churn is a meaningful fraction of the 12 MB git pack (~12 of the
  top-20 blobs are 250–293 KB versions of it). Splitting the monolith also shrinks history churn.

### 2.2 Chapter YAML schema drift + no validation (HIGH)
- ch00 uses `player_units:`; ch01 `deploy_limit`+`deploy_slots`; ch02 both *and* a `deployment:`
  block; ch03–08 only `deployment:`. `introduces:` appears in 4 of 9 chapters. **No two built
  chapters express their roster the same way** — invisible only because each chapter has a
  hand-written injector.
- Only gates: `check_yaml_parses` (parse-only) + `check_chapter_status`. **No schema/shape check.**
- Fix together with 2.1: normalize the deployment representation, add a schema check to
  `check.py`, and make the generic injector consume it.

### 2.3 Documentation entropy (MEDIUM — cheap to fix, compounds if ignored)
Contradictions & staleness found:
- **Cast count**: `CLAUDE.md:3` says 7 PCs; roster is 8 (`pcs/`, HANDOFF, CLASSES.md);
  `PRD.md:162` + `decisions.md:1247` say "10 cast". Pick one number, define "cast".
- **Dead issue refs**: battle anims are **#65**, but `PRD.md:86`, `CREDITS.md:23,25`, and
  `_parked/meesmickle-kitsune-fox/PARKED.md` still cite #39 — PARKED.md also claims "the inserter
  isn't built yet / post-MVP", flatly contradicted by shipped RBG+braulo anims.
- **roadmap.md:90** lists Baxby joining Ch2; he's recruited in ch01 YAML (CHAPTERS.md agrees).
- **`lore/README.md:44-47`** cites a dead `/tmp/genlore.rb` generator and "pyyaml is not
  installed" — contradicts `decisions.md:24` (Ruby retired 06-09).
- **`decisions.md:453`** flags a roadmap line as stale that has already been fixed.
- **Dead scaffolding contradicting a locked ADR**: `engine/{d20-combat,spell-slots,damage-types,
  class-defs,ui}`, `data/homebrew/*`, `data/srd/` contain only `.gitkeep`, while `decisions.md:30`
  says stock FE8 classes / no SRD pull. The *real* engine layer is `tools/inject/`. Delete the
  stubs (and update CLAUDE.md's "new engine modules go in `engine/`" line to point at the real seam).
- **HANDOFF.md is ~35% live state, ~65% durable reference**: the Gotchas section (durable rules →
  decisions.md / tool docstrings), the Wolfram prompt pack (→ `pcs/wolfram.yaml` or
  `lore/wolfram.md`), tools quick-ref (→ docstrings/CLAUDE.md), and workflow/release sections that
  duplicate CLAUDE.md/decisions.md. A "live state" doc that needs 200 lines re-read every session
  start is a per-session context tax.
- **CLAUDE.md is no longer lean**: the Design-placement-test and feature-flow sections are
  long-form rationale duplicating `decisions.md` §Coordination (both even say so). The
  hardcoded macOS base-ROM path also appears in `tools/setup-toolchain.sh:17`; and
  `tools/render_reskin_concepts.py:7` has a *different, dead* absolute path.
- **decisions.md** (1,744 lines): navigable thanks to dated ADRs, but §Combat (~680 lines)
  restates chunks of `rules-mapping.md` — the exact duplication its own §Documentation Model
  forbids. Needs a TOC.
- No `memory.md` exists — HANDOFF.md is the de-facto memory file; the findings above are about it.

### 2.4 Process overhead & thin review (MEDIUM)
- **~10 of the last 25 PRs are `docs(handoff)` bookkeeping**, self-merged in 1–4 minutes. That's
  full PR ceremony (branch, PR, CI, squash, cleanup) for session-state notes. Handoff-only changes
  should bypass the PR machinery (direct docs-only push, or fold into the feature PR they describe).
- **Median time-to-merge is 3.8 minutes with zero GitHub-visible reviews** — `/code-review`
  happens (PR bodies mention fixes) but leaves no artifact. Use `--comment` so findings live on
  the PR.
- **Branch cleanup isn't happening**: 8+ stale post-merge branches despite the flow mandating
  deletion; #103 (the cleanup punch list) is itself unmerged. Enable GitHub's
  "Automatically delete head branches" and this problem disappears permanently.
- `demo/ch2-gifs` is a long-lived unmerged branch holding a stale demo reel — decide
  regenerate-or-drop.

### 2.5 Backlog hygiene (LOW-MEDIUM)
- 35 open issues; **19 stale >2 weeks**, 15 untouched since 2026-05-31 (mostly legitimate
  future-chapter placeholders, but untriaged for a month).
- **#39 duplicates #65** (close/fold). **#56** is a junk playtest-form submission (close).
  **#49** may be superseded by `docs/roadmap.md`.

### 2.6 Test skew + CI speed (LOW-MEDIUM)
- Coverage is strong on *pure* logic (difficulty 557 L, combat 244 L, build helpers 376 L) and
  near-zero on the **injectors — the biggest, most-changed code**. Mitigated by fail-fast guards +
  the Lua harness, but the `sys.exit`-everywhere style (92 in build_campaign.py) is what makes
  them untestable; the 2.1 refactor should return errors instead.
- Injection **ordering constraints are comments, not code** (`inject_ch01` before
  `inject_prologue`, lord-floor after lord-select). One reorder breaks the build;
  `check_engine_guards_present` checks presence, not order.
- **CI rebuilds agbcc from source every run** (~2–3 min of the 3.2-min median). One
  `actions/cache` step roughly halves CI.
- mGBA playtest *scenarios* are not CI-gated (known/accepted — worth a periodic manual cadence).

### 2.7 Git/binary bloat (LOW)
- Pack is 12 MB (healthy), but **`docs/demo/` = 9.2 MB of GIF/PNG**, with `braulo-anim.gif`
  committed twice (~7 MB of history). The mobile-delivery workaround is legitimate — but optimize
  GIFs before committing (palette-reduce/resize ≈ 5–10× smaller), and consider an orphan
  `demo-assets` branch so main's history stops accreting superseded binaries.
- `data/portraits/wolfram.jpeg` (1.6 MB) and `meesmickle.jpeg` (816 KB) are unoptimized reference
  JPEGs.

## 4. Benchmark vs. beacon projects

- **Engine-base tier** (pret/pokeemerald, RH-Hideout's pokeemerald-expansion: semver releases,
  CONTRIBUTING, headless battle-test runner in CI): manchego-stars matches the shape of this tier
  — and your mGBA memory-asserted harness is the same *kind* of asset as expansion's test runner,
  which is the only mature automated-gameplay-testing example in the entire scene.
- **Celebrated finished FE hacks** (Vision Quest, Souls of the Forest, The Last Promise): built in
  FEBuilder with **no git, no CI, no automated tests, no docs** — their QA is ~16-person human
  playtester rosters, Discord feedback, and multi-year incremental public demos. You exceed every
  shipped FE hack on engineering. They beat you on: player contact cadence and finishing.
- **Field norms**: git = minority practice (buildfile/decomp niche); CI for a *game* (not an
  engine base) = rare; automated gameplay tests = very rare; data-driven chapter pipelines = rare
  (buildfiles' CSV tables are the closest analog); docs-as-source-of-truth with drift gates =
  essentially unheard of.
- **AI-agent precedents**: FE Infinity (one-shot LLM chapter generation over buildfiles, little
  QA) and Agent Oak (Claude on pokeemerald: modify→build-to-verify→memory journal→issue-driven
  direction). Your loop — YAML data layer + fail-fast injection + in-emulator assertions +
  handoff/ADR memory — is more rigorous than both.

**Calibration:** on process/infrastructure you are top-of-field, ahead of every *shipped* FE hack
we could find. The risk profile is the opposite of the typical hack: they die from chaos; the
failure mode to guard against here is infrastructure/documentation absorbing the hours that
chapters need. The monolith refactor (§2.1) is the one piece of engineering that directly buys
content velocity; most other findings are janitorial. After that, the winning move the beacons
teach is: ship chapter batches to your players early and often.
